# AGENTS.md instructions

## 通用规则

- 默认使用简体中文回复。
- 当前仓库使用 `skills/<skill-name>/` 管理个人 Codex skills。
- 不要在 skill 目录内新增 README、安装指南、变更日志等辅助文档；只保留 `SKILL.md` 和执行该 skill 必需的资源。
- 新增或修改 skill 后，运行 `./scripts/validate-all.sh`，并在最终说明中记录验证结果；如无法运行，说明阻塞原因、未覆盖风险和已做的替代检查。
- 安装到本机 Codex 时使用 `./scripts/install.sh`。
- 不要新增依赖；脚本优先使用系统自带工具或仓库已有运行时。涉及结构化数据、状态机或测试时，优先使用已有解析 / 测试能力，避免脆弱字符串匹配。

## Skill 校验与门禁边界

- 设计或修改 skill 的校验 / 门禁脚本时，让 agent 在边界内先做判断，而不是机械套流程规则：在变更说明、review 结论或提交说明中说明目标属于结构格式校验、流程状态门禁、边界检查还是内容审查，记录判断依据，并判断是否适合机器校验。
- 只有检查目标可确定、可复现、无需解释业务内容时，才写进脚本；语义判断、证据强度判断、防伪绑定或“看起来可信”的启发式判断应交给 reviewer / 人工审查。
- 如果用户要求把语义判断、证据强度判断、防伪绑定或“看起来可信”写进脚本，先停止直接实现，说明不可机器校验的原因，并给出可机器校验的替代协议，例如固定 token、结构化字段、明确终态或人工 reviewer 结论引用。
- 机器只检查协议是否闭合，不替人作判断：过程态可以通过普通结构校验，但最终门禁只能接受对应协议定义的明确终态；脚本可以要求结论已被显式记录，不能根据内容、规模、风险或证据强度推断结论是否正确。
- 涉及校验 / 门禁脚本的最终说明至少记录：校验目标类型、机器可判定依据、明确不检查的语义范围、验证命令和结果。
- skill 中的脚本如需测试，测试文件放在仓库级 `tests/<skill-name>/` 下；不要放进 `skills/<skill-name>/`，也不要在 skill 文档里描述仓库级测试路径。
