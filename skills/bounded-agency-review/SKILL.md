# Bounded Agency Review

## 使命

审查一份 AI skill / 规则 / AGENTS.md / workflow / prompt 是否把 AI 设计成：

> 在清晰边界内，根据目标函数做选择，并为选择留下证据和审计痕迹的 agent。

不要把目标理解成“让规则更多、更细、更硬”。

本 skill 审查的是 **AI 行为约束系统本身**，不是业务代码实现。除非用户明确要求，否则不要对代码 diff 做实现级 review。

## 核心主旨

审查时始终回到 agent contract 的三句主旨：

1. **给 AI 选择权，但不给它无责任的自由。**
2. **给 AI 判断空间，但要求它留下证据。**
3. **用边界约束它，用目标函数驱动它，用审计机制信任它。**

这三句是判定标准，不是口号：AI 可以在边界内选择路径，但必须能说明目标函数、证据、风险和审计痕迹。

因此，审查重点不是“有没有规定每一步怎么做”，而是：

- AI 的目标是否明确；
- AI 的硬边界是否清楚；
- AI 是否有足够的判断空间；
- 多个方案冲突时是否有目标函数 / 优先级；
- 重要选择是否要求 evidence；
- 高风险、不确定、越界时是否会升级或停止；
- 低风险任务是否允许压缩流程；
- 审计记录是否能支持后续复盘；
- 规则是否制造了假控制、表演流程或表格主义。

## 适用范围

使用本 skill 审查：

- `SKILL.md`
- `AGENTS.md`
- agent rules / 编辑器规则文件
- 分阶段执行 / 规格变更 / 审查门禁类工作流规则
- prompt instructions
- subagent protocol
- review protocol
- planning / audit / evidence 文件规则
- AI coding policy / execution contract

不要自动用于：

- 普通代码 review；
- lint / type / test 失败分析；
- 产品需求评审；
- 文案润色；
- 用户只是问某条规则怎么理解。

如果用户要创建、安装、调试或实现某个 skill / workflow，优先使用对应的 creator / implementation / debug skill；只有当用户明确要求审查、改进或重写 agent contract 时，再使用本 skill 做元审查。

如果用户只给了很少内容，先基于已给文本做 best-effort review；不要因为缺少完整仓库而拒绝。

## 审查原则

### P1. 规则必须先分类，再评价

先对关键规则或高影响章节归类：

| 类型 | 含义 | 合理形态 |
| --- | --- | --- |
| 硬边界 | 永远不能越过的安全 / 范围 / 语义边界 | 少、清晰、可判定 |
| 目标函数 | 冲突时如何权衡好坏 | 有排序，有例外处理 |
| 启发式 | 默认倾向，但可被证据推翻 | 明确“默认 / 通常 / 优先” |
| 决策协议 | AI 如何比较方案、记录选择、处理不确定 | 轻量但强制 evidence |
| 风险门禁 | 不同风险等级下的执行权限 | 风险改变行为 |
| 审计格式 | durable evidence / decision log / status | 低冗余、可追踪 |
| 流程动作 | 读文件、写计划、跑测试、提交等动作 | 与风险和 evidence 绑定 |
| 输出格式 | 最终报告 / review 表格 / status 字段 | 只保留能提高审计性的字段 |

不要把所有“must”都当成硬边界。很多 must 实际应该是风险门禁、启发式或 evidence obligation。

### P2. 审查选择空间，不奖励机械流程

优先找这些问题：

- 规则是否把 AI 变成“遇到 X 就做 Y”的流程执行器；
- AI 是否能比较多个方案；
- AI 是否能说明“为什么不是另一个方案”；
- AI 是否能在启发式不适用时说明偏离原因；
- AI 是否被允许先诊断，再决定执行路径；
- AI 是否被迫执行与风险不匹配的重流程。

好规则不应该替 AI 做完所有判断。好规则应该约束判断过程。

### P3. 审查责任机制，不信任空泛承诺

所有重要 claim 都应能落到 evidence。

常见 claim：

- “需求已理解”；
- “没有分叉”；
- “没有越界”；
- “当前方案最小”；
- “测试覆盖了回归”；
- “风险等级已判定”；
- “可以自动执行”；
- “review passed”；
- “done”。

审查这些 claim 是否要求：

- 代码事实；
- 测试结果；
- 类型检查 / lint 结果；
- 调用链证据；
- diff 边界；
- 设计文档 / 决策记录 / 规范引用；
- 审查材料证据；
- 明确的 unknown / residual risk。

没有 evidence 的 done 是高风险信号。

### P3.5. 区分 evidence obligation 和 evidence carrier

审查时不要默认把 evidence 等同于脚本输出、固定字段或 durable artifact。

先判断规则是否声明了关键 claim 需要 evidence，再判断当前 evidence 载体是否足以支撑对应风险。

Evidence carrier 可以是：

- 会话说明；
- final report；
- 人审 review；
- 脚本输出；
- review package；
- durable artifact。

载体选择取决于被审对象的风险等级、执行场景和是否需要跨任务复用。低风险或人工闭环场景里，能由 AI 会话描述并由人 review 复核的点，不应直接判为 skill 契约缺口或脚本缺口。

只有在这些情况下，才把 evidence 问题列为 finding：

- 规则要求关键 claim，但没有要求说明依据；
- 当前载体不足以支撑对应风险；
- 需要跨任务复用、独立恢复或自动门禁，却只留在易丢失的会话里；
- claim 会放行高风险状态，但没有失败 / cannot-verify 语义；
- 被审对象已经声明需要机器门禁，而当前协议无法稳定判定。

### P4. 审查边界，而不是增加束缚

硬边界应该保护：

- 用户授权；
- 任务范围；
- 公共 API / 契约；
- 数据语义；
- 安全 / 权限 / 隐私；
- 测试信号；
- 提交边界；
- 外发 / 发布 / push / merge 等不可逆动作。

如果硬边界太多、太宽、太模糊，会导致 AI 不敢判断，只会暂停或填表。此时应建议：

- 缩窄为可判定边界；
- 改成风险分层；
- 改成目标函数；
- 改成启发式；
- 改成 evidence requirement。

### P5. 审查目标函数

当多个原则冲突时，规则必须说明怎么取舍。

建议优先级基线：

1. 正确性；
2. 用户意图；
3. 可验证性；
4. 任务范围控制；
5. 最小必要改动；
6. 可维护性；
7. 与现有模式一致；
8. 性能，除非性能就是任务目标；
9. 表达 / 格式一致性。

重点审查：

- “最小改动”是否可能压过正确性；
- “遵循现有风格”是否可能延续坏抽象；
- “必须补测试”是否可能产生无意义测试；
- “不要问用户”是否可能导致擅自产品决策；
- “遇到不确定就问”是否可能导致 AI 放弃可自行查证的判断。

### P6. 审查风险分层是否真的改变行为

风险标签必须有执行后果。

推荐行为模型：

| 风险 | AI 权限 | 必要责任 |
| --- | --- | --- |
| A / 低风险 | 可自动执行 | 简短上下文判断 + 相关验证 |
| B / 中风险 | 可自动执行，但必须可审计 | 上下文预检 + evidence + review |
| C / 高风险 | 不可直接实现 | 方案 / 风险 / evidence 后等待确认 |
| 严重 | 不执行 | 拒绝、升级或要求人工裁决 |

如果规则有风险标签，但不同风险等级都跑同样流程，这是形式主义。
如果高风险也能自动执行，这是无责任自由。

### P7. 审查审计机制是否有真值

好的审计机制记录：

- 稳定结论；
- 支撑证据；
- 决策分叉；
- 风险变化；
- 验证结果；
- 未覆盖风险；
- 选择依据；
- 偏离默认规则的理由。

坏审计机制记录：

- 用户说了什么的流水；
- agent 做了什么的流水；
- “已理解 / 已检查 / 已完成”之类不可验证句子；
- 重复粘贴 plan / diff；
- 为了格式而格式化；
- 不改变后续执行边界的琐碎结论。

审计记录的判断标准：

> 删除这条记录后，是否会影响后续边界、风险、验证、责任归属或复盘能力？

如果不会，通常不该记录为 durable audit。

### P8. 严格识别假控制

假控制常见形态：

- 固定标题很多，但没有实质 evidence；
- 每阶段都有 status，但 status 不阻塞任何行为；
- review 只复述 diff，不给 failed / cannot-verify；
- “必须跑测试”，但不要求说明测试覆盖了什么；
- “必须创建计划”，但计划不影响执行边界；
- “必须确认”，但确认语义可被“继续 / 好的”跨阶段污染；
- “不要越界”，但没有允许 / 禁止修改边界；
- “AI 自查”，但没有外部 evidence 或独立 reviewer 输入。

对假控制要直接指出，并建议删除、合并或改成 evidence-backed gate。

## 审查流程

### 步骤 1：确认审查对象和使用场景

先判断被审查文本是什么：

- 单个 skill；
- 多个 skill；
- AGENTS.md；
- 项目 rules；
- execution workflow；
- review protocol；
- planning / audit 文件规范；
- subagent protocol。

同时判断它服务的任务类型：

- 轻量任务；
- 中大型 coding task；
- 高风险契约 / 状态机 / 权限 / 数据任务；
- 文档 / 规格 / 变更提案；
- review-only；
- commit / PR / CI 工作流。

如果文本没有声明适用范围，列为问题。

### 步骤 1.5：声明材料覆盖

在给出 `通过-边界健康`、`通过-需小修`、`可直接使用`、`PASS`、`没有明显问题` 等强结论前，必须先声明本次审查覆盖范围。

至少说明：

- 已读取的文件、章节或片段；
- 明确已知但未读取或未覆盖的文件；
- 因缺失、过大、不可访问、未提供而无法检查的材料；
- 本次结论是否只基于局部材料。

如果对象是一个 skill 目录，至少检查：

- `SKILL.md`；
- frontmatter / description；
- `SKILL.md` 中显式引用的脚本、模板、示例、测试或规则文件；
- 与触发、执行、审查、输出直接相关的关键文件。

如果无法确认是否读全，必须标记为 `partial-coverage`。
如果关键材料缺失或不可访问，必须标记为 `cannot-verify`，并且不得给出 `PASS`、`可直接启用`、`没有明显问题` 等强结论；只能给出基于已读材料的局部判断和下一步需要补读的材料。

### 步骤 2：生成契约地图

把规则系统映射成：

```md
目标：
- <AI 要优化什么>

硬边界：
- <绝对不能越过什么>

决策优先级：
- <冲突时怎么取舍>

选择空间：
- <AI 可以选择什么>
- <AI 不能选择什么>

证据义务：
- <哪些 claim 要证据>

风险 / 升级：
- <何时自动，何时确认，何时停止>

审计产物：
- <哪些记录是真源，记录什么，不记录什么>

压缩规则：
- <低风险如何跳过重流程>
```

这个契约地图用来发现缺口、重叠和错位。

### 步骤 3：做规则分类

至少挑关键章节 / 规则做分类表。

格式：

```md
| 章节 / 规则 | 当前类型 | 建议类型 | 诊断 |
| --- | --- | --- | --- |
| <规则> | 硬边界 / 工作流 / 启发式 / 不清楚 | <建议类型> | <为什么> |
```

不要逐字分类整篇长文；优先覆盖高影响规则。

### 步骤 4：找问题

每条问题必须包含：

- 严重性；
- 类型；
- 证据：引用具体路径 / 行号 / 章节；如果问题来自缺失内容，说明已检查哪些相关章节但未找到；
- 问题；
- 行为影响；
- 建议；
- 建议改写，除非不需要改写。

严重性：

- `阻塞`：会导致越权、错误执行、错误信任、无法审计；
- `高`：显著降低判断质量或制造假控制；
- `中`：局部过硬、过松、重复或缺少证据；
- `低`：表达、结构、成本优化。

类型：

- `缺少边界`
- `过度控制`
- `无边界自由`
- `缺少目标函数`
- `缺少证据`
- `风险分层缺口`
- `升级缺口`
- `审计缺口`
- `形式主义`
- `token 成本`
- `触发范围`
- `状态语义`
- `表演式 review`
- `确认语义`

### 步骤 5：给最小可用补丁

不要默认重写整份规则。优先给：

1. 保留什么；
2. 改哪几条；
3. 哪些强制要求降级成启发式；
4. 哪些模糊规则改成证据义务；
5. 哪些工作流改成风险分层；
6. 哪些状态需要阻塞语义；
7. 哪些审计记录要删掉或合并。

除非用户明确要求“直接重写”，否则输出定向补丁而不是全文重写。

## 结论枚举

使用下列结论之一：

- `通过-边界健康`：边界、目标函数、选择空间、evidence、审计都健康；只需小修。
- `通过-需小修`：方向正确，有少量过硬 / 过细 / 缺证据处。
- `混合-过度控制`：安全和审计强，但规则过多、流程过硬，AI 判断空间被压缩。
- `混合-边界不足`：有选择空间，但边界、evidence 或升级条件不足。
- `失败-流程执行器`：主要把 AI 变成机械流程执行器，选择空间和目标函数不足。
- `失败-无责任自由`：给了 AI 太多自由，但缺边界、证据、审计和升级。

结论必须带执行后果：

- `通过-边界健康`：可直接使用，只需可选小修。
- `通过-需小修`：可使用，但建议在下一次编辑中修复非阻塞问题。
- `混合-过度控制` / `混合-边界不足`：不建议作为强 gate 或团队默认规则依赖；先修高严重性问题。
- `失败-流程执行器` / `失败-无责任自由`：不应安装、推广或作为自动化 gate 的依据。

## 输出格式

先根据审查对象大小和风险选择输出深度：

- 低风险：保留结论、最大风险、3-5 条问题、建议的下一个补丁。
- 中风险：增加契约地图和关键规则分类。
- 高风险 / 严重 / 用户明确要求完整审查：使用完整输出。

所有输出都必须说明审查范围和未覆盖内容。选择压缩输出时，可以压缩为一句材料覆盖声明，但不能省略 `partial-coverage` 或 `cannot-verify` 状态。

完整输出：

````markdown
# 边界化自主审查

## 材料覆盖
- 已读取: <文件 / 章节 / 片段>
- 未覆盖: <明确未读或不在本轮范围内的材料>
- 无法检查: <缺失 / 过大 / 不可访问 / 未提供的材料；无则写“无”>
- 覆盖状态: full-coverage / partial-coverage / cannot-verify
````

## 改写模式库

优先给定向改写，不要重写整份规则。常用改写方向：

| 问题形态 | 改写方向 |
| --- | --- |
| “必须优先最小改动。” | 改成目标函数：正确性和验证先满足，再选最小必要改动；偏离时说明证据和剩余风险。 |
| “必须补测试。” | 改成证据义务：关键行为变更必须有验证证据；无法测试时说明原因和替代证据。 |
| “所有任务都必须生成完整计划、审计文件和审查材料。” | 改成风险分层：低风险可轻量，中风险要可审计，高风险先方案和风险后确认。 |
| “不要修改无关代码。” | 改成可判定边界：声明允许 / 禁止修改的文件、目录或契约；越界前先停止并更新边界。 |
| “完成后说明已完成。” | 改成声明 / 证据 / 状态：声明可观察结果、支撑证据、验证状态和未覆盖风险。 |
| “必须沿用现有模式。” | 改成可反驳启发式：默认沿用；若会延续错误语义、重复或不可验证，可带证据偏离。 |
| “完成后做独立审查。” | 改成拒收门禁：审查必须输出通过 / 失败 / 无法验证；失败和未解决的无法验证阻塞 done。 |
| “用户确认后继续。” | 改成确认边界：确认只适用于刚提出的单一问题；澄清确认、执行授权和范围扩大不能互相替代。 |

## 专门检查项：skill 触发范围

审查 skill 时必须检查 frontmatter / description：

- 是否过宽，导致普通任务也触发重流程；
- 是否过窄，导致高风险场景漏触发；
- 是否写明“不要在普通代码任务中自动触发”；
- 是否描述了与其他 skills 的边界；
- 是否说明更具体 skill 与本 skill 的优先级。

触发范围本身也是边界。触发过宽会制造流程污染；触发过窄会造成安全缺口。

## 专门检查项：与分阶段执行类工作流的关系

如果审查分阶段执行、计划文件、审查材料、完成门禁类规则，特别检查：

- 阶段 / 子任务是否按可观察业务结果，而不是代码动作；
- 分叉记录是否只记录真正会约束后续执行的决策；
- 审计文件是否只记录长证据，而不是过程流水；
- 审查材料是否有拒收能力，而不是只做总结；
- 风险等级是否改变执行权限；
- 固定口令是否保护授权语义，而不是纯仪式；
- 完成态是否被硬门禁和独立审查阻塞；
- 范围化提交是否受 diff 边界和验证状态约束；
- 低风险任务是否有轻量档，不被完整流程拖死；
- 高风险任务是否只允许方案 + 证据 + 人工确认，不自动越权实现。

## 禁止事项

- 不要因为规则多就给高分。
- 不要默认建议“再加一条 must”。
- 不要把个人偏好的流程格式包装成硬边界。
- 不要要求所有任务都走完整流程。
- 不要接受无 evidence 的 passed / done。
- 不要把“AI 必须遵循规则”作为最终目标；最终目标是 bounded agency。
- 不要把缺少信息当作立刻问用户；先判断是否可由代码、文件、历史记录或工具查证。
- 不要输出只有抽象原则、没有定向改写的 review。

## 最终检查

在给出 review 前，问自己：

1. 我是否区分了硬边界、目标函数、启发式、流程和审计格式？
2. 我是否指出了 AI 可以在哪里选择、哪里不能选择？
3. 我是否检查了重要 claim 的 evidence obligation？
4. 我是否检查了风险分层是否改变执行行为？
5. 我是否识别了假控制和形式主义？
6. 我是否给了最小可用补丁，而不是泛泛建议？
7. 我是否声明了材料覆盖范围，并在覆盖不足时避免强结论？
8. 我的建议是否会让 AI 更会判断，而不是更会填表？
9. 我是否把“可由 AI 说明 + 人 review 复核闭环”的事项误判成了必须新增字段、文件或脚本门禁？
