## Claim 与 evidence 示例

常见 claim：

- “需求已理解”；
- “没有分叉”；
- “没有越界”；
- “当前方案最小”；
- “测试覆盖了回归”；
- “风险等级已判定”；
- “可以自动执行”；
- “review passed”；
- “done”。

可支撑这些 claim 的 evidence 包括：

- 代码事实；
- 测试结果；
- 类型检查 / lint 结果；
- 调用链证据；
- diff 边界；
- 设计文档 / 决策记录 / 规范引用；
- 审查材料证据；
- 明确的 unknown / residual risk。

## 审计机制与假控制示例

好的审计机制记录：

- 稳定结论；
- 支撑证据；
- 决策分叉；
- 风险变化；
- 验证结果；
- 未覆盖风险；
- 选择依据；
- 偏离默认规则的理由。

坏审计机制记录：

- 用户说了什么的流水；
- agent 做了什么的流水；
- “已理解 / 已检查 / 已完成”之类不可验证句子；
- 重复粘贴 plan / diff；
- 为了格式而格式化；
- 不改变后续执行边界的琐碎结论。

假控制常见形态：

- 固定标题很多，但没有实质 evidence；
- 每阶段都有 status，但 status 不阻塞任何行为；
- review 只复述 diff，不给 failed / cannot-verify；
- “必须跑测试”，但不要求说明测试覆盖了什么；
- “必须创建计划”，但计划不影响执行边界；
- “必须确认”，但确认语义可被“继续 / 好的”跨阶段污染；
- “不要越界”，但没有允许 / 禁止修改边界；
- “AI 自查”，但没有外部 evidence 或独立 reviewer 输入。

## 目标函数冲突检查

- 审查重点是否变成“有没有规定每一步怎么做”；
- 规则是否把 AI 变成“遇到 X 就做 Y”的流程执行器；
- “最小改动”是否可能压过正确性；
- “遵循现有风格”是否可能延续坏抽象；
- “必须补测试”是否可能产生无意义测试；
- “不要问用户”是否可能导致擅自产品决策；
- “遇到不确定就问”是否可能导致 AI 放弃可自行查证的判断。

## 定向改写模式

优先给定向改写，不要重写整份规则。

| 问题形态 | 改写方向 |
| --- | --- |
| “必须优先最小改动。” | 改成目标函数：正确性和验证先满足，再选最小必要改动；偏离时说明证据和剩余风险。 |
| “必须补测试。” | 改成证据义务：关键行为变更必须有验证证据；无法测试时说明原因和替代证据。 |
| “所有任务都必须生成完整计划、审计文件和审查材料。” | 改成风险分层：低风险可轻量，中风险要可审计，高风险先方案和风险后确认。 |
| “不要修改无关代码。” | 改成可判定边界：声明允许 / 禁止修改的文件、目录或契约；越界前先停止并更新边界。 |
| “完成后说明已完成。” | 改成声明 / 证据 / 状态：声明可观察结果、支撑证据、验证状态和未覆盖风险。 |
| “必须沿用现有模式。” | 改成可反驳启发式：默认沿用；若会延续错误语义、重复或不可验证，可带证据偏离。 |
| “完成后做独立审查。” | 改成拒收门禁：审查必须输出通过 / 失败 / 无法验证；失败和未解决的无法验证阻塞 done。 |
| “用户确认后继续。” | 改成确认边界：确认只适用于刚提出的单一问题；澄清确认、执行授权和范围扩大不能互相替代。 |

## Skill 触发范围检查

审查 skill 时检查 frontmatter / description：

- 是否过宽，导致普通任务也触发重流程；
- 是否过窄，导致高风险场景漏触发；
- 是否写明“不要在普通代码任务中自动触发”；
- 是否描述了与其他 skills 的边界；
- 是否说明更具体 skill 与本 skill 的优先级。

触发范围本身也是边界。触发过宽会制造流程污染；触发过窄会造成安全缺口。

## 分阶段执行类工作流检查

如果审查分阶段执行、计划文件、审查材料、完成门禁类规则，特别检查：

- 阶段 / 子任务是否按可观察业务结果，而不是代码动作；
- 分叉记录是否只记录真正会约束后续执行的决策；
- 审计文件是否只记录长证据，而不是过程流水；
- 审查材料是否有拒收能力，而不是只做总结；
- 风险等级是否改变执行权限；
- 固定口令是否保护授权语义，而不是纯仪式；
- 完成态是否被硬门禁和独立审查阻塞；
- 范围化提交是否受 diff 边界和验证状态约束；
- 低风险任务是否有轻量档，不被完整流程拖死；
- 高风险任务是否只允许方案 + 证据 + 人工确认，不自动越权实现。
