# 切片开发

## 第一原则

先清分叉，再确认执行。

- **决策分叉**：需求、行为、范围、术语、上游、验证或记录语义不唯一。用户回答只确认口径，不授权执行。
- **拷问选择**：两类门禁触发——拆任务层在切片清单产出后问整体拆分，执行分片层在候选需确认 / 跨模块片审查前问该片；一律默认“推荐：拷问”。必须要求用户若要作出门禁选择，只回复 `拷问` 或 `不拷问`；只有去掉首尾空白后完全等于这两个口令之一才有效。`是 / 确认 / 继续 / 好的` 等普通确认词无效，必须重问，不能结合上下文推断。用户提出问题、补充事实或指出流程问题时，先回答 / 处理，门禁状态不推进，再重问同一门禁。
- **拷问收口**：用户进入拷问后，不能因为答完一个问题就自动结束拷问。无更多拷问问题时，必须发“拷问收口候选”，并告知用户可回复 `结束拷问` / `继续拷问`，也可以直接提出其他问题；只有 `结束拷问` 有效后，才可按拷问产出的明确结论写回；若只压实选项但未形成结论，转入单点分叉确认。用户回复 `继续拷问`、提出问题、补充事实或指出流程问题时，先回答 / 处理，保持 `grilling`，再重问同一收口门禁。
- **执行确认**：切片无开放分叉但命中高风险。发执行预告并问“确认执行切片 N 吗？”。用户确认只授权控制器按当前 task brief 派发 implementer subagent；若确认后未派发即中断，续跑时必须重新预告和确认。
- **分叉审查阶段**：每个切片执行前必须先单独证明“没有决策分叉”。审查阶段禁止发执行预告、禁止请求执行确认、禁止展开实现方案。
- **全局约束**：本次任务所有切片都必须继承、不可违反的全局需求约束，写入 `plan.md` 顶层 `## 全局约束`；执行和 AI Review 必须读取。
- **计划一致性预检**：切片清单产出后、拆分拷问门禁前，先检查整份计划内部是否一致；只写顶部 `计划一致性预检` 状态，分叉进 `decisions.md`，长矩阵进 `audits.md`，不新增 plan 章节。
- **上下文预检**：动手前先暴露需要理解的上下文、必读上下文、项目规范、允许 / 禁止修改、非目标和停止条件；读完上下文前不得实现。写 `ready` 时必填字段不得仍是占位内容，`项目规范` 和 `禁止修改` 可显式写 `无`。完整档写回当前切片块，轻量档在会话输出短版。
- **拒收门禁**：实现后必须跑硬门禁和 package-first AI Review；每个切片最多自动修复两次，次数用尽仍失败则停止。
- **Claims / Evidence / Status**：完整档每个执行切片用 `claims/<S-id>.json` 记录可验证声明、证据和状态；Claim 是执行约束，不是事后总结。控制器按证据判断 claim 状态；脚本只校验 JSON / 字段形状，`close-check` 只信 claims 的最终状态。
- **交接文件**：完整档当前片自动实现前、或需确认片执行预告前，生成 `task-briefs/<S-id>.md` 作为 implementer subagent 的窄上下文入口；brief 会渲染当前片 claims。implementer 只按 task brief 与其必读上下文实现，完成后必须填写 `task-reports/<S-id>.json` 结构化报告，在 `claimUpdates` 中提出状态建议和证据；只有 `conclusion: ready-for-review` 才能生成 review-package。legacy `task-reports/<S-id>.md` 只作兼容读取。
- **审查包**：完整档每片硬门禁后、标记 done 前生成 `review-packages/<S-id>.md` 临时输入；生成前必须先通过 `validate`，脚本会维护 `dev-plans/.gitignore` 里的 `*/review-packages/**`、`*/task-briefs/**`、`*/task-reports/**`；reviewer subagent 以该 package 为主输入，先审 Claims，再输出 `需求符合性` / `切片边界 / 交接一致性` / `代码质量 / AI 污染检查` 三 verdict；AI Review 结论表使用 `Evidence` / `Note` 五列格式，Evidence 填写 review-package 章节名、文件路径或固定不适用标记；脚本只校验固定 verdict、枚举和 Evidence 非空；审计真源写回 plan / D/A，生成文件不进 diff-check 或 package inventory。
- **产物角色边界**：`plan.md` 是任务状态、边界、风险和执行位置真源；`claims/<S-id>.json` 是声明 / 证据 / 状态真源；task brief、task report、review-package 和 whole-review-package 是注意力收束视图，不是真源。收束视图限制 implementer / reviewer 的默认注意力；当 P0/P1 claim、边界、证据或 verdict 无法从视图判断时，必须 focused 回源到代码、测试、diff 或真源状态，并把证据写回 plan / D/A / claims，不能用 package 完整性替代事实判断。
- **用户验收**：AI Review 通过后按 [PLAN-FILE.md](PLAN-FILE.md) 的 `用户验收` 字段收口；自动片默认不逐片停下，需确认片 / C 类片必须停下给用户验收。
- **整任务审查**：默认不启用。用户明确要求整体验收 / 跨切片审查 / 发布就绪度，或控制器 / reviewer 判断单切片 AI Review 覆盖不了全局约束、跨切片交接、非目标回归、需求闭合或残余风险时，才运行 `whole-review-package` 并在 plan 顶部添加 `整任务审查：package-generated` 与 `## 整任务审查结论`；最终只允许写 `passed` / `blocked`，`blocked` 和 `package-generated` 都阻塞 `close-check`。
- **最终收口**：`close-check` 是最终格式门禁，不只读状态字段；它会检查终态、diff-check gate evidence、claims JSON 形状、task brief、ready-for-review task report、slice review-package 格式和已启用的整任务审查包；report 里的 `proposedStatus` 不会被当作 claim 终态。
- **切片提交**：每个切片完成验证、硬门禁、AI Review 和用户验收后，自动 scoped commit 本片**代码**修改；`dev-plans` 记录不随片提交，默认收口时独立 commit，用户中途要求时也可独立提交，plan 的提交永远独立于代码 commit。提交边界见 [EXECUTION-RULES.md](EXECUTION-RULES.md)。

发现决策分叉时，禁止发执行确认；执行确认不能替代需求澄清。用户回答“继续 / 保持 / 确认”时，只按刚刚提出的问题解释，不能跨阶段当作分叉结论或执行授权；若刚刚提出的是拷问选择，这类普通确认词无效，必须重问固定口令；若刚刚提出的是拷问收口，只有 `结束拷问` 会收口，`继续拷问` 或直接提问都保持拷问态。

## 档位

- **轻量**：当前 context 内可完成、全为「自动」片，且不需要跨 context 恢复、subagent 隔离、独立 review-package、逐 claim 证据闭环或 close-check（如单文件 / 文案 / 局部样式 / 明确小 bug）→ 先在会话输出短版上下文预检，再直接做 + 修改后验证；B 类必跑 AI Review；不切片、不记录、不走门禁状态机、不自动 commit。判出 C、命中「需确认」面或同一任务第二次越界时，停下转完整档。细则见 [EXECUTION-RULES.md](EXECUTION-RULES.md) 的「轻量档执行」。
- **完整**：命中任一控制需求时走完整流程：用户显式发起 `/sliced-dev`、续跑已有 `dev-plans`、含任一「需确认」/ C 类 / 高风险契约片、预计跨 context、需要 subagent 隔离实现、需要独立 review-package、需要逐 claim 证据闭环或 close-check。单纯“中大型 / 跨模块”不是充分条件；只有它导致上述控制需求时才进入完整档。

## 拷问分层

拷问分两层，互不替代，各管各的产出物：

- **拆任务层**：针对“任务怎么理解、整体怎么切”。清单成型**前**的需求 / 范围 / 术语 / 上游口径分叉，走「分叉处理协议」的拷问选择；清单成型**后**的整体拆分结构，走「拆分拷问门禁」。
- **执行分片层**：针对“选定的某一片具体怎么做、有没有分叉”。候选需确认 / 跨模块片的拷问入口是「切片前拷问门禁」；低风险候选自动片可跳过该门禁，但若审查首次发现分叉，必须先补问该片拷问选择。已问过该片拷问门禁后，审查阶段发现分叉才直接进分叉确认，不重问拷问。

## 分叉处理协议

任务级门禁、切片、切片前分叉审查发现决策分叉时，共用查证、写回、重跑这套骨架，不在各阶段各写一套问法：

1. 先查证，收敛成一个最小分叉；能靠代码、文件、git 状态或上游材料解决的不要问用户。
2. 写入 `decisions.md` 的 `D*`，只记录“状态 / 关联 / 证据 / 推荐 / 问题”；这里的“问题”是后续可能要问的分叉结论，不是立即抛给用户；长证据用 `A*` 写入 `audits.md`。
3. 是否先问拷问选择，按所在层决定：
   - **拆任务层**（任务级门禁 / 切片）：停下问拷问选择“是否先进行拷问？推荐：拷问。若要作出门禁选择，请回复：拷问 / 不拷问。”。
   - **执行分片层**（切片前分叉审查）：若该片拷问门禁已问过，不重问拷问，直接进第 5 步；若低风险候选自动片跳过门禁后首次发现分叉，先补问该片拷问选择。
4. 用户回复 `拷问` 时，按「拷问执行方式」选择工具压实分叉；进入拷问后必须经过收口门禁，只有 `结束拷问` 会收口，`继续拷问` 或直接提问都保持拷问态。收口后若已形成唯一结论，进入第 6 步写回；若只压实选项但未形成结论，进入第 5 步。
5. 用户回复 `不拷问`、处于不重问拷问的执行分片层，或拷问收口后仍未形成唯一结论时，才按已记录的“问题”问一个分叉结论，并给推荐答案。
6. 写回结论后，把 D 状态改为 `decided` 并同步相关切片关联项，再回到触发该分叉的阶段重跑门禁（切片阶段发现的分叉回到任务级分叉门禁）；分叉未清零前，不切片、不判风险、不发执行预告。

## 拷问执行方式

用户选择拷问后，优先使用已有拷问 skill，不用普通问答替代：

1. 用户明确指定 `grill-me`、`grill-with-docs` 或其他审查方式时，按用户指定执行。
2. 未指定但需要同步写回文档、计划、上游依据、ADR 或语义记录时，优先使用 `grill-with-docs`。
3. 未指定且只需要压实设计 / 需求 / 行为分叉时，优先使用 `grill-me`。
4. 只有这两个 skill 不适用或不可用时，才使用当前 skill 内的普通拷问追问。

拷问执行期间：

- 每轮只问一个具体问题；用户回答只解释为该问题的回答，不代表拷问结束、分叉清零或授权执行。
- 还存在未压实问题时，继续下一轮拷问。
- 进入拷问后，若当前入口有对应门禁字段，把 `拆分拷问` / 切片 `门禁` 写为 `grilling`。
- 判断没有更多问题时，保持 `grilling`，输出“拷问收口候选”，列出已压实结论和剩余风险，然后停下问：“是否结束拷问？可回复：结束拷问 / 继续拷问；如果还有其他问题，也可以直接提问。”
- 用户回复 `结束拷问` 后，按当前入口收口：拆分拷问写 `拆分拷问：grilled` 并进入首个切片流程；切片前拷问写该片 `门禁：grilled` 并进入切片前分叉审查；分叉处理里的拷问若已有唯一结论则写回 D，否则转入单点分叉确认。
- 用户回复 `继续拷问` 后，保持 `grilling`，继续拷问。
- 用户直接提出问题、补充事实或指出流程问题时，先回答 / 处理，保持 `grilling`，再重问收口门禁。
- 用户回复其他无明确含义内容时，不更新门禁状态、不进入分叉审查或执行确认，只重问收口门禁。

## 拆分拷问门禁

切片清单产出并通过计划一致性预检后、进入第一个切片前，对整体拆分方案问一次（完整档触发，轻量档不问；整个任务只问一次）：“是否先对整体拆分方案进行拷问？推荐：拷问。若要作出门禁选择，请回复：拷问 / 不拷问。”

- 对象是成型拆分方案的**结构**：切片边界、片间依赖、候选标签是否合理；不重问清单成型前已处理的口径分叉。
- 这是审查方式选择，不是分叉结论，也不是执行授权。
- 用户回复 `拷问` 时，按「拷问执行方式」压实拆分；通过 `结束拷问 / 继续拷问` 收口门禁后，把 `拆分拷问` 写为 `grilled`，结论写回 `dev-plans` 的切片清单，再进入首个切片流程。
- 用户回复 `不拷问` 时，把 `拆分拷问` 写为 `no-grill`，再进入首个切片流程；若该片命中「切片前拷问门禁」，按该门禁处理，低风险候选自动片可跳过。
- 用户回复不是 `拷问` 或 `不拷问` 时，不更新门禁状态、不进入下一阶段；若用户在提问、补充事实或指出流程问题，先回答 / 处理，再重问同一个拷问选择。
- 若清单成型前的拷问已覆盖整体结构，agent 可标注原因，但仍要求用户回复 `不拷问` 后才跳过本门禁。

## 切片前拷问门禁

选定候选需确认片、跨模块片或用户要求的切片后，进入切片前分叉审查前问一次：“是否先进行拷问？推荐：拷问。若要作出门禁选择，请回复：拷问 / 不拷问。”这是执行分片层拷问的**唯一入口**，发生在 agent 判定“无分叉”之前。

- 这是审查方式选择，不是分叉结论，也不是执行授权。
- 用户回复 `拷问` 时，按「拷问执行方式」先压实边界；通过 `结束拷问 / 继续拷问` 收口门禁后，把该切片 `门禁` 写为 `grilled`，结论写回 `dev-plans`，再进入切片前分叉审查。
- 用户回复 `不拷问` 时，把该切片 `门禁` 写为 `no-grill`，才由 agent 进入切片前分叉审查并输出无分叉 / 发现分叉结论。
- 用户回复不是 `拷问` 或 `不拷问` 时，不更新门禁状态、不进入分叉审查或执行确认；若用户在提问、补充事实或指出流程问题，先回答 / 处理，再重问同一个拷问选择。
- 已问过本门禁的切片，审查阶段发现分叉时直接进分叉确认（按「分叉处理协议」问分叉结论），不再重问拷问。
- 低风险候选自动片可跳过本门禁；若维护 `dev-plans`，把该片 `门禁` 写为 `not-applicable`。随后直接输出简短分叉审查；若审查首次发现分叉，先补问该片拷问选择。

## 完整流程

1. **开场审计**
   - 读已有 `dev-plans/`、`git status --short -uall`、当前切片代码和上游。
   - 完整档续跑优先用 `roster` 看切片概览、`show current` 加载计划头与当前切片块，而不是整篇读 `plan.md`；关联项中的 D/A 仍按规则读取正文（命令见 [SCRIPTS.md](SCRIPTS.md)）。
   - 同时扫描 `decisions.md` 中仍为 `open` 的 D；若已产出切片但 open D 未挂到任一切片 `关联项`，先修记录或回到任务级分叉门禁，不能直接续跑当前切片。
   - 识别 `计划一致性预检`、`拆分拷问` 与各片 `门禁` 状态；计划一致性预检未 `passed` 时，不能进入拆分拷问或执行。
   - `当前切片` 只能指向可执行切片；若指向 `split` 父项、`done` / `skipped` 终态、`待定` 或不存在的 ID，先修记录再继续。
   - 报告档位和是否维护 `dev-plans`。
2. **任务级分叉门禁**
   - 判断任务级 `任务内容`、影响范围、排除范围、上游和验收是否足够进入切片。
   - 还要判断是否存在会改变切片边界、验证方式或风险标签的未确认点。
   - 能靠搜索、读代码、检查文件 / git 状态解决的先查；不要把可查问题抛给用户。
   - 仍有任务级分叉时，调用「分叉处理协议」；任务级分叉清零前不进入切片。
3. **切片**
   - 只细化已确认目标，不用切片补需求口径。
   - 每片端到端、可独立验证；记录 `任务内容`、验收、依赖、候选标签和执行控制字段。
   - 稳定跨切片输入 / 输出只写当前切片可选 `#### 切片交接`；不建立顶层接口注册表。普通 `依赖：S*` 不强制触发交接小节。
   - 按业务结果拆，不按代码形状拆；不要把切片写成“抽 helper / 加 fallback / 新公共 utils / normalize”。
   - 若切片时发现上游模糊、冲突、遗漏，或切片边界 / 验证 / 风险标签不唯一，调用「分叉处理协议」并回到任务级分叉门禁。
   - 若执行前或执行中发现单片过大导致边界或验证不清，先停下建议拆小；范围、验证或风险标签变化时重新进入分叉门禁。
   - 切片阶段只给候选标签；正式 `自动 / 需确认` 结论必须在切片前分叉审查通过后判定。
   - 候选标签：
     - **候选自动**：预期低风险，后续可自动做 + 自动验证。
     - **候选需确认**：预期命中风险，后续需要用户批准。
4. **计划一致性预检**
   - 切片清单产出后、拆分拷问门禁前执行；检查总目标 / 非目标、全局约束、术语、旧行为覆盖、片间依赖、切片交接、候选风险、验证口径、D/A 关联和依赖图是否互相一致。
   - 无问题时写 `计划一致性预检：passed`；发现需要用户拍板的冲突时写 `blocked（D*）` 并按「分叉处理协议」处理；长证据或矩阵写 `A*`。
   - 预检是门禁动作，不是 plan 章节；禁止把检查流水、问答过程或完整矩阵写进 `plan.md`。
5. **拆分拷问门禁**
   - `计划一致性预检：passed` 后、进首片前，对整体拆分方案问一次（完整档；整个任务一次），详见「拆分拷问门禁」。
   - 清单成型前拷问已覆盖整体结构时，可标注原因，但仍要求用户回复 `不拷问` 后才跳过。
6. **切片前分叉审查**
   - 选定当前切片后，先停在「分叉审查阶段」；候选需确认片 / 跨模块片先通过「切片前拷问门禁」。
   - 即使当前切片 `关联项` 为空，也必须按当前代码和上游重判。
   - 输出形态、审查项、发现分叉后的回流见 [EXECUTION-RULES.md](EXECUTION-RULES.md)。
   - 审查无分叉后，进入上下文预检。
7. **上下文预检与 Claims**
   - 只更新当前切片 `#### 上下文预检`、执行控制字段和 `claims/<S-id>.json`，不改业务代码。
   - 写明风险 A/B/C、必读上下文、项目规范、允许修改、禁止修改、非目标、停止条件，并核对 `全局约束` 与当前切片 `切片交接`。
   - 读完 must-read 后再更新 `上下文预检：ready`；`需理解`、`必读上下文`、`允许修改`、`非目标`、`停止条件` 不得仍是 `待执行前补充`、`TBD`、`TODO`、`待补充`、`未填写` 等占位内容；`项目规范` 和 `禁止修改` 可写 `无`；若上下文不足或需要越界，写 `blocked` 并停止。
   - 生成 task brief 前创建或细化 `claims/<S-id>.json`；claims 至少覆盖行为、边界、验证和残余风险中的关键项。
   - C 类切片在输出方案后必须停下等人工确认；方案并入执行预告，方案确认与执行确认是同一次停顿。
8. **执行与门禁**
   - 自动片：准备 claims → 生成 task-brief → 派发 implementer subagent → 接收门禁 → 硬门禁并更新 claims evidence/status → 生成 review-package → reviewer subagent 三 verdict → 有限修复 / 补证 → 按 `PLAN-FILE.md` 写回自动片用户验收状态 → 代码 scoped commit → 报告 → 选下一片；若用户明确要求逐片验收，则写 `用户验收：pending` 并停下。
   - 需确认片：确认无开放分叉 → 准备 claims → 生成 task-brief → 发引用该 brief 的执行预告 → 等确认 → 派发 implementer subagent → 接收门禁 → 硬门禁并更新 claims evidence/status → 生成 review-package → reviewer subagent 三 verdict → 有限修复 / 补证 → `用户验收：pending` 停下 → 用户验收通过后代码 scoped commit → 报告 → 选下一片。若执行预告后 task brief 变化，必须重新预告并重新确认；若确认后未派发即中断，续跑时也必须重新预告并重新确认。
   - `AI Review：passed` 必须有完整三 verdict，且不得出现 `failed`、`cannot-verify-from-package` 或 `critical`。任一阻塞项存在、claims 未全部 `verified` / `waived`、用户验收未达到 [PLAN-FILE.md](PLAN-FILE.md) 的完成态约束、自动修复次数用尽仍有门禁失败、风险升为 C、或需要修改禁止修改 / 非允许修改范围时，不能把切片标 `done`，必须停止并报告。
9. **收口**
   - 普通任务收口前运行 `close-check`；它会重新检查终态、diff-check gate evidence、claims JSON 形状、task brief / task report / review-package 和已启用的整任务审查包，不接受只手写 passed 状态。
   - 需要整任务审查的任务由控制器生成 `whole-review-package` 做跨切片审查，并把 `整任务审查` 先更新为 `package-generated`；审查通过后写 `整任务审查：passed` 和固定五 verdict 表，阻塞时写 `整任务审查：blocked`，Evidence 写 review-package 章节或文件路径，阻塞说明写正文。
   - 高风险任务提示转 `rules-review deep / cross-slice`，不得静默当成自动通过。
   - 收口时把 `dev-plans` 记录独立 commit（默认唯一一次 plan 提交，独立于代码 commit）。
   - 验证命令、失败 / 跳过说明、完成报告和 `dev-plans` 提交边界见 [EXECUTION-RULES.md](EXECUTION-RULES.md)。

## 分叉速查

命中任一条 = 存在决策分叉，按「分叉处理协议」处理，不发执行预告：

- 旧实现有可观察行为，新计划未明确是否纳入。
- 旧实现的可观察行为没有追到调用链、异步等待、失败传播或副作用边界。
- 一个词可能指向不同业务概念、入口、宿主、页面或能力。
- request、类型、状态、路由、入口、公共导出或文件落点存在多个合理位置。
- 同一句失败语义可能对应不同可观察结果（如局部 reject、阻断入口初始化、静默降级）。
- 上游文档、计划、全局约束与当前代码冲突。
- 影响范围 / 排除范围无法具体到文件、目录或契约。
- 验证方式无法证明完成，或会改变风险标签。
- 为覆盖一个局部行为，测试被迫 mock 大量无关模块或 import 聚合入口，测试落点不唯一。
- `dev-plans` 的记录状态影响跨会话审计语义。

## 记录

命中完整档控制需求时维护目录化 `dev-plans/`；否则只用会话 Task 列表。完整档默认结构：

```text
dev-plans/
  .gitignore
  YYYY-MM-DD-<slug>/
    plan.md
    decisions.md
    audits.md
    claims/          # 每片结构化 Claim / Evidence / Status 真源，提交入库
      S1.json
    task-briefs/      # 生成文件，gitignore
    task-reports/     # implementer 结构化交付报告，默认 S*.json，gitignore
    review-packages/  # 生成文件，gitignore
```

新建完整档优先使用 `node <sliced-dev-skill-dir>/scripts/dev-plan.mjs init ...`；结构或状态修改后、切片提交前运行 `validate`。

记录只分三类：状态机过程、证据、分叉结论。状态机过程只写当前状态、计划一致性预检状态、门禁状态、切片状态和下一步；证据写代码事实、调用链、矩阵和命令结果；分叉结论只写会约束后续执行的稳定口径。禁止把“用户回复了什么、什么时候 validate、什么时候进入 / 结束拷问、agent 做了什么动作”写成分叉结论或审计结论；这些是过程，只能由状态字段、门禁字段、验证摘要或会话回复承载。判据：一条结论删掉后若不改变后续边界、范围、风险或验证方式，就不写成结论。
